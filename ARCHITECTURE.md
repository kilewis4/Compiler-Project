## Compilation Pipeline

```
source.unn
    ↓
[LEXER] → tokens → [PARSER] → AST → [CODEGEN] → executable
```

### Phase 1: Lexical Analysis (Lexer)
- **Input**: Source text
- **Output**: Token stream
- **What it does**: 
  - Recognizes keywords (`exit`)
  - Recognizes literals (integers)
  - Recognizes separators (`;`, `(`, `)`)
  - Skips whitespace, tracks line/column positions
  
**Files**: `lexer.c`, `lexer.h`

### Phase 2: Syntax Analysis (Parser)
- **Input**: Token stream
- **Output**: Abstract Syntax Tree (AST)
- **What it does**:
  - Validates token sequence against grammar rules
  - Builds hierarchical tree structure
  - Provides error messages for malformed code

**Files**: `parser.c`, `parser.h`

### Phase 3: Code Generation (NOT YET IMPLEMENTED)
- **Input**: AST
- **Output**: Executable code (assembly, C, bytecode, etc.)
- **What it does**: Converts tree structure to machine instructions

## Fixes Made to Code

| Issue | Old Code | Fixed |
|-------|----------|-------|
| **strcmp logic** | `if(strcmp(keyword, "exit"))` | `if(strcmp(keyword, "exit") == 0)` |
| **Null termination** | No null terminator added | `value[index] = '\0'` |
| **Double character read** | Characters were read twice | Added `ungetc()` to put back unread chars |
| **File handling** | File never closed | Added proper `fclose()` |
| **Memory leaks** | Scattered malloc/free | Centralized in token/stream functions |
| **Error handling** | None | Added null checks and error messages |

## Current Language Grammar

```
PROGRAM     → STATEMENT*
STATEMENT   → EXIT_STMT
EXIT_STMT   → "exit" "(" EXPR ")" ";"
EXPR        → INTEGER
INTEGER     → [0-9]+
```

**In English**: A program is zero or more statements. Currently, the only statement is `exit()` with an integer argument.

## SHORT TERM: Next Features to Add (Easy)

### 1. Add More Keywords
**Effort**: 30 minutes
- Add `return`, `if`, `while`, `for` keywords
- Location: Update `TokenTypeKeyword` enum in `token.h`
- Update `lexer_keyword()` function in `lexer.c`

### 2. Add Operators
**Effort**: 1 hour
- Add `+`, `-`, `*`, `/`, `==`, `!=` operators
- Location: Create new token type `TokenTypeOperator` in `token.h`
- Update lexer to recognize operators

### 3. Expression Support
**Effort**: 1.5 hours
- Allow `exit(5 + 3);` not just `exit(5);`
- Update `EXPR` grammar rule: `EXPR → EXPR "+" EXPR | INTEGER`
- Implement operator precedence in parser

### 4. String Literals
**Effort**: 45 minutes
- Support `"hello"` strings
- Add `TOKEN_STRING` type
- Update lexer to handle quoted strings

### 5. Basic Code Generation
**Effort**: 2 hours  
**Option A - Interpreter** (Simplest):
```c
int interpret_ast(ASTNode *node) {
    if (node->type == AST_STATEMENT_EXIT) {
        int exit_code = interpret_ast(node->data.exit_stmt.expr);
        exit(exit_code);
    }
    if (node->type == AST_EXPR_INT) {
        return node->data.int_expr.value;
    }
}
```

**Option B - Generate C code** (Medium):
```c
void codegen_c(ASTNode *ast, FILE *out) {
    fprintf(out, "#include <stdlib.h>\n");
    fprintf(out, "int main() {\n");
    // Walk AST and emit C code
    fprintf(out, "}\n");
}
```

## MEDIUM TERM: More Complex Features (Intermediate)

### 1. Variable Declarations
**Effort**: 3 hours
```unn
int x = 5;
exit(x);
```

**Implementation**:
- Add `TokenTypeDataType` enum (`INT`, `FLOAT`, `STRING`)
- Create symbol table to track variables
- Update parser grammar: `STATEMENT → VAR_DECL | EXIT_STMT`
- Add type checking pass

### 2. Functions
**Effort**: 4 hours
```unn
int add(int a, int b) {
    exit(a + b);
}
```

**Implementation**:
- Add `TokenKeyword: FUNCTION, RETURN`
- Update AST with `AST_FUNCTION_DEF` and `AST_FUNCTION_CALL` nodes
- Implement function scope tracking
- Symbol table for function signatures

### 3. Control Flow
**Effort**: 3 hours
```unn
if (x > 0) {
    exit(1);
} else {
    exit(0);
}
```

**Implementation**:
- Add comparison operators: `>`, `<`, `>=`, `<=`
- Add boolean literals: `true`, `false`
- AST nodes: `AST_IF_STMT`, `AST_WHILE_STMT`, `AST_FOR_STMT`
- Parser recursively parses block statements `{ ... }`

### 4. Type System & Checking
**Effort**: 4 hours
- Semantic analysis pass (after parsing, before codegen)
- Check: function arguments match parameters
- Check: return values match declared types
- Check: variable types match usage

### 5. Symbol Table
**Effort**: 2 hours
- Track variables and their types in scope
- Detect undefined variable usage
- Detect duplicate declarations

## LONG TERM: Advanced Features (Expert)

### 1. Advanced Expressions
- Ternary operator: `condition ? true_val : false_val`
- Logical operators: `&&`, `||`, `!`
- Bitwise operators: `&`, `|`, `^`, `<<`, `>>`

### 2. Pointers & References
```unn
int *ptr = &x;
int value = *ptr;
```
- Pointer types: `int*`, `float**`
- Address-of: `&`
- Dereference: `*`

### 3. Arrays & Structs
```unn
int arr[10];
struct Point { int x; int y; };
```

### 4. Register Allocation & Optimization
- Map AST operations to CPU registers
- Optimize away redundant operations
- Dead code elimination

### 5. Backend Code Generation
**Option A - x86-64 Assembly**:
```asm
mov rdi, 5
call exit
```

**Option B - LLVM IR** (Intermediate Representation):
```
define i32 @main() {
  call void @exit(i32 5)
}
```

**Option C - WebAssembly**:
```wasm
(module
  (func $exit (param i32))
  (export "main" (func $main))
  (func $main
    i32.const 5
    call $exit))
```

### 6. Garbage Collection
- Automatic memory management
- Reference counting or mark-and-sweep

### 7. Module System
- Import/export declarations
- Link multiple files
- Standard library (printf, malloc, etc.)

## References for Compiler Theory

1. **Lexing**: Regular expressions define valid tokens
2. **Parsing**: Context-free grammar defines valid syntax
3. **AST**: Tree representation of program structure
4. **Codegen**: Transform AST to target language/architecture
5. **Optimization**: Improve code before final output

The classic textbook is "Compilers: Principles, Techniques, and Tools" (the "Dragon Book").
